Engineering Philosophy

"돌아가게 만드는 것" 과 "운영할 수 있게 만드는 것" 은 다르다고 생각합니다.

파이프라인 하나를 만들더라도 다음 사람이 이어서 쓸 수 있는지, 새벽에 장애가 나면 어떻게 재처리할 수 있는지를 먼저 고민합니다. 연구·분석 인력이 제각각 서버에서 중복 데이터를 쓰던 흐름을 공용 환경·공용 저장으로 정리한 것도 같은 기준에서 나왔습니다. 메인 역량은 데이터 파이프라인 설계·구현·운영이며, 이 경험을 바탕으로 플랫폼 구조를 함께 고민하고 있습니다.

데이터 플랫폼 구조 검토

Structure Review
인라이플 · 2025~2026
On-Prem Kubernetes 기반 Data Platform Structure

ML/DL·분석이 제각각 서버에 흩어져 동일 데이터가 중복되고, ETL / Query / ML 워크로드가 한 노드에 몰리는 문제를 먼저 짚었습니다.

이를 Storage–Compute 분리와 레이어·엔진 분리로 풀기 위해 구조를 설계했고, 사내 Kubernetes에 JupyterHub, Airflow on Kubernetes, MinIO, MySQL 메타 DB 등을 구성해 설계안과 실제 구성을 검증했습니다.
핵심 컴포넌트는 운영 중이며, Trino·GPU·로그 집계는 검토 단계입니다.

Raw / Processed / Serving 단위로 책임을 나누고, 온프레·향후 클라우드 확장이 가능한 방향으로 정리합니다.

① 레이어 분리 (책임 명확화)
Raw / Processed / Serving로 나눠 계층별 책임과 변경 범위를 분리
② 확장성 고려 (Multi-Engine)
온프레 환경에 맞춰 Storage–Compute를 분리하고, Spark 중심 ETL 기준의 확장 구조를 검토
③ 운영 안정성 기준
GPU 워크로드 독립 실행(검토)으로 리소스 경합을 줄이고, overwrite 기반 멱등성으로 운영 안정성을 우선
계층별 검토안 (7 Layers)
Platform Control
Kubernetes — JupyterHub, Airflow on K8s(DAG·Spark Job 트리거), 팀 공용 환경
Data Ingestion
Spark Batch · Kafka→Structured Streaming(Parquet, 시간당 대규모) · 시간 파티션 증분 · Iceberg 적재 구조
Storage & Meta
MinIO (S3 호환·Distributed) + Iceberg (ACID·Schema Evolution·Time Travel) + Hive Metastore (단일 Catalog → Multi-Engine 정합성)
Compute
Spark (대규모 ETL·Feature Table 생성) 중심 + Trino (검토: 향후 Iceberg 직접 조회·Ad-hoc SQL·BI 확장)
ML Compute
GPU Node Pool 독립 실행(검토) · PyIceberg/Arrow → Polars (고속 전처리) → PyTorch (CUDA) 스택
Serving
Batch Inference 결과 제공 · 외부 시스템 연계 · BI/Analytics Query 지원
Observability
Prometheus + Grafana (메트릭·리소스·Pod 상태) + Log Aggregation (검토: Loki/ELK) — 장애 원인 분석 지원
Kubernetes Node Pool 분리 검토안
Control Pool · K8s Control Plane Storage Pool · MinIO ETL Pool · Spark Worker Query Pool · Trino Worker (검토) GPU Pool · ML Workload (검토) Infra Pool · Monitoring
트레이드오프 분석
선택 대안 결정 근거
Iceberg Delta Lake 벤더 종속 최소화, Multi-Engine 지원, Open Standard
MinIO HDFS 온프레미스 환경에서 비용 효율적인 분산 스토리지 구성을 위해 선택 · Erasure Coding 기반 데이터 복구 및 장애 대응 구조 확보 · S3 호환 API 기반 운영·연동 표준 확보
Spark ETL Python ETL Airflow는 오케스트레이션에 집중하고, 실제 처리 계층은 Spark로 분리. 파티션 기반 병렬 처리와 재시도·복구로 대용량 배치·증분 적재를 안정적으로 운영
ML → Spark 미사용 Spark ML ML 팀 기술 스택 존중, Polars 기반 고속 전처리 적합
Batch 중심 Streaming 현 요구사항(배치 중심 집계·DW 적재)에 최적화 · 향후 스트리밍 처리가 필요할 경우 확장 가능한 구조로 설계
구조 검토 포인트
